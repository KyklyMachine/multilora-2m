# Постановка задачи: Multi-LoRA Backbone Inference

> Efficient single-pass shared backbone с fan-out на N LoRA-адаптеров

---

## 1. Контекст

Есть мультимодальная модель **Qwen3-VL-8B** (36 трансформерных блоков, `hidden_size = 4096`).
Модель используется как **эмбеддер**: на каждый входящий запрос (изображение + текст) генерируется ровно 1 токен, и в качестве результата берётся скрытое состояние последнего слоя после pooling.

К модели обучено **20–30 LoRA-адаптеров**, каждый из которых применяется к **последним 8 слоям** (слои 28–35). Каждый запрос должен пройти через **все** адаптеры одновременно, возвращая отдельный эмбеддинг для каждого.

---

## 2. Проблема

### Наивный подход

При стандартном multi-LoRA serving каждый адаптер требует отдельного forward pass через всю модель:

```
1 запрос × 30 адаптеров = 30 полных прогонов через все 36 слоёв
```

Слои 0–27 **идентичны** для всех адаптеров — LoRA там не применяется. Их пересчёт 30 раз является избыточным.

**Следствие:** эффективный throughput системы деградирует как `RPS / N`, где N — количество адаптеров.

### Целевой подход

```
Слои 0–27  →  считаются 1 раз
Слои 28–35 →  считаются N раз параллельно (в одном батче)
```

Вычислительная нагрузка перестаёт линейно зависеть от числа адаптеров.

---

## 3. Цель

Построить систему инференса, которая для каждого входящего запроса:

1. Прогоняет общую часть модели (слои 0–27) **ровно один раз**.
2. Передаёт скрытые состояния в финальные слои (28–35).
3. Применяет все N LoRA-адаптеров **параллельно** в одном батче.
4. Возвращает N эмбеддингов — по одному на каждый адаптер.

---

## 4. Архитектура системы

### Компоненты

```
┌─────────────┐     tokens      ┌──────────────────────────────────────┐
│             │ ──────────────► │            BACKBONE                  │
│ Orchestrator│                 │  Vision Encoder + MLP merger         │
│             │                 │  LLM layers 0–27  (frozen, no LoRA)  │
│  (единая    │                 │                                      │
│   точка     │                 │  Выход: [seq_len, 4096] float16      │
│   входа)    │                 └──────────────┬───────────────────────┘
│             │                                │  [seq_len, 4096]
│             │                 ┌──────────────▼───────────────────────┐
│             │                 │           LoRA HEADS                 │
│             │                 │  expand × N → [N, seq_len, 4096]    │
│             │                 │  Layers 28–35 + LoRA ∆W              │
│             │                 │  (base weights shared, ∆W уникален) │
│             │                 │  Pooling last token                  │
│             │                 │                                      │
│             │ ◄────────────── │  Выход: [N, 4096]                   │
└─────────────┘   [N, 4096]     └──────────────────────────────────────┘
```

### Компонент 1 — Backbone

| | |
|---|---|
| **Вход** | изображение + текст |
| **Содержимое** | Vision Encoder → MLP merger → LLM layers 0–27 |
| **Веса** | заморожены, LoRA не применяется |
| **Выход** | `[seq_len, 4096]` float16 — скрытые состояния **всех токенов** со слоя 27 |
| **Запусков на запрос** | 1 |

> **Важно:** на выходе backbone — полный sequence `[seq_len, 4096]`, а не только последний токен `[4096]`. Это необходимо, потому что MHA в слоях 28–35 вычисляет attention по всей последовательности. Pooling делается **после** прохода через финальные слои.

### Компонент 2 — LoRA Heads

| | |
|---|---|
| **Вход** | `[seq_len, 4096]` от Backbone |
| **Содержимое** | expand × N → LLM layers 28–35 с LoRA → pooling last token |
| **Веса** | базовые веса слоёв 28–35 shared; матрицы A, B уникальны для каждого адаптера |
| **Вычисление** | батчевый GEMM: `x·W + x·Aᵢ·Bᵢ` для всех i одновременно |
| **Выход** | `[N, 4096]` — N эмбеддингов |

### Компонент 3 — Orchestrator

Единая точка входа. Принимает запрос от клиента, последовательно вызывает Backbone → LoRA Heads, возвращает N эмбеддингов. При батчинге входящих запросов — управляет очередью.

## 5. Поток данных

```
Client
  │  image + text
  ▼
Orchestrator
  │  tokens  [seq_len]
  ▼
Backbone (layers 0–27)
  │  [seq_len, 4096] float16
  │  ≈ seq_len × 4096 × 2 байта  (при seq_len=1024 → ~8 MB)
  ▼
LoRA Heads (layers 28–35)
  │  expand: [seq_len, 4096] → [N, seq_len, 4096]
  │  forward: [N, seq_len, 4096] через 8 слоёв с N разными LoRA
  │  pooling: [N, seq_len, 4096] → [N, 4096]
  ▼
Orchestrator
  │  [N, 4096]
  ▼
Client
```

---

## 6. Требования

### Функциональные

- Вход: изображение + текст (мультимодальный запрос).
- Выход: N эмбеддингов `[4096]`, по одному на каждый адаптер.
- Каждый запрос проходит через **все** N адаптеров.
- Backbone вычисляется **ровно один раз** на запрос.
- LoRA применяется **только к слоям 28–35**.
- Pooling (взятие последнего токена) производится **после** слоя 35.
- Горячая замена и добавление адаптеров без остановки сервиса.

### Нефункциональные

- Latency: приемлемая для продакшена; целевой ориентир — не хуже нескольких последовательных запусков полной модели (точный бенчмарк определяется по результатам замеров прототипа).
- Throughput: не деградирует линейно с ростом числа адаптеров.
- Память: базовые веса слоёв 28–35 хранятся в **одном экземпляре**.
- Корректность: результаты численно идентичны полному forward pass с LoRA.
- Передача тензора `[seq_len, 4096]` между backbone и heads — минимальная задержка.
- Возможность батчинга входящих запросов на уровне backbone.

---

## 7. Ограничения и допущения

- **Модель:** Qwen3-VL-8B, 36 трансформерных блоков, `hidden_size = 4096`.
- **Split point:** после слоя 27. Backbone = слои 0–27, Heads = слои 28–35.
- **Математическая корректность split:** LoRA-адаптеры обучены стандартным образом на слоях 28–35. При разделении модели в этой точке distribution shift отсутствует — слои 28–35 во время обучения получали тот же `h₂₇`, что и при раздельном инференсе.
- **Количество адаптеров:** 20–30, все загружены в VRAM одновременно.
- **Режим работы:** embedding-only (`max_tokens=1`). Autoregressive генерация не требуется.
- **Инфраструктура:** одна или несколько GPU-нод; backbone и heads могут располагаться на одном устройстве.

## 8. Смежные работы

Задача не имеет готовой open-source реализации. Ближайшие академические работы:

| Работа | Связь с задачей |
|---|---|
| [PrefillShare](https://arxiv.org/abs/2602.12029) (NAVER Cloud, 2026) | Ближайший аналог: shared prefill module + task-specific decode modules поверх vLLM. Нет публичного кода. |
| [Activated LoRA](https://arxiv.org/abs/2504.12397) (IBM, 2026) | KV-cache reuse при **последовательной** смене адаптеров. Не решает fan-out. |
| [InfiniLoRA](https://arxiv.org/abs/2604.07173) (2026) | Disaggregated LoRA serving. Масштабирование кластера, не single-request fan-out. |
| [S-LoRA](https://arxiv.org/abs/2311.03285) / [Punica](https://arxiv.org/abs/2310.18547) | Батчевый GEMM для разных адаптеров (SGMV kernel) — применим для компонента LoRA Heads. |

## 9. Варианты реализации

### Вариант 1 — Два сервиса + shared memory

Backbone и LoRA Heads — отдельные процессы на одной ноде. Тензор `[seq_len, 4096]` передаётся через shared memory / CUDA IPC (без сериализации и сетевого стека).

**Плюсы:** чистое разделение ответственности, backbone масштабируется независимо, проще горячая замена адаптеров.  
**Минусы:** требуется координация памяти между процессами.

### Вариант 2 — Единый кастомный inference-сервер

Один процесс, загружающий всю модель. Forward pass разделён внутри: backbone → expand → batched heads → pool. Нет никакого межпроцессного взаимодействия.

**Плюсы:** минимальная latency, нет оверхеда на передачу тензоров.  
**Минусы:** backbone и heads нельзя масштабировать независимо; сложнее встроить в существующий inference-фреймворк.

### Вариант 3 — Disaggregated serving (по мотивам PrefillShare)

Backbone на выделенном prefill-узле, LoRA Heads на decode-узлах. Тензор `[seq_len, 4096]` передаётся по сети через RDMA / NVLink. Позволяет горизонтально масштабировать backbone и heads под разную нагрузку.

**Плюсы:** горизонтальное масштабирование, наилучший throughput под высокой нагрузкой.  
**Минусы:** наибольшая инфраструктурная сложность, сетевая latency на передачу тензора.
